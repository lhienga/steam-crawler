\r\n\t\t\t\t\tAn excellent way to learn about circuit logic and computer architecture. There are several difficulty spikes that occur, some in places where some in-between puzzles are likely needed, and some where the puzzle prompt simply does not provide enough information to get started easily. If you have some experience programming you can likely get pretty far without having to resort to reading Wikipedia, but otherwise expect an uphill battle that may require you consult some external resources.

By the end of the game you are given significant freedom in regards to how you design your computer architecture, with the game asking you to implement features without specifying the exact opcodes for you. I really liked this aspect of the game and it encouraged me to get out a pen and paper and think about how I wanted my computer to work.

The final set of assembly puzzles made me really yearn for a more robust assembler. Using "_" for zeroed bytes everywhere works, but can makes things messy. And having to manually enter register/immediate variants for every instruction was a hassle (especially after a re-architecture where I needed to re-enter all of them).

I really look forward to coming back to this game in the future when there are more puzzles and quality of life improvements have been made (eg. wire colouring is a pain, and can reduce visibility of bytes, viewing memory when debugging is difficult, monitored sources cannot be renamed, etc.).

\r\n\t\t\t\t\tWhen I purchased this game, I was hoping to bridge a conceptual gap in my understanding of computers. I had a basic understanding of logic gates, and I had a basic understanding of assembly language. However, I didn't understand how an assortment of logic gates could be arranged in a way to create a CPU, and how that CPU actually performed the instructions defined in assembly language.

This game answered that question for me, and for that I will automatically recommend it. It was challenging to get there, and the game doesn't hold your hand along the way, but it is a challenging topic so I expected that.

That said, I do have quite a bit of constructive criticism as of the time of this review (it is early access). If I remember, I'll update my review if/when these points are addressed. I'm pretty forgetful though, so don't assume that they are still issues just because they are still in my review.

Stability
  • The game crashes somewhat frequently for me, probably once every hour or two. This occurs most often when copying, pasting, undoing, and redoing operations that affect multiple components/wires, so crash frequency depends on how often I do that. Fortunately, the game automatically saves everything, so this has never resulted in any lost progress, but I do lose my undo context as well as clipboard contents, which can be annoying

UX
  • The level view should have tooltips that show the name/description of completed levels. I frequently wanted to refer back to concepts described in previous levels, but finding them was a pain (had to go into each level individually)
  • I'd like to be able to zoom out further. Perhaps it's because of my aspect ratio (32:9), but I could rarely see an entire schematic at one time, and felt there was plenty of room to zoom out without reducing legibility
  • The "Instructions" modal is very helpful, but I'd like a way to easily see what the upcoming test/expected outputs are. This is done well in the earlier levels, but not so well in the CPU architecture section.
  • The icons in the top-left aren't very intuitive. Here are my complaints and suggestions regarding those:
    1. The "Main Menu" button is a hamburger icon, which makes me think it would be a pause menu. It instead takes you directly to the main menu when you click it
    2. The "Switch Schematic" icon looks like a "Save" button, but it functions like an "Open" button. Consider using a Folder icon instead, since that is more traditionally associated with "Open"
    3. The "Instructions" icon uses a cog icon, which is typically used for settings/options menus. Consider using the bit shape (teardrop shape with corner oriented to the bottom left) as the icon instead. I didn't have as much issue misclicking this one though
    4. The "Component Factory" uses a wrench icon, which is also typically used for settings/options menus
    5. The "Level Map" looks as much like a circuit as it does the level map, so I frequently click that one expecting to go to the component factory.
    6. Consider making the hamburger icon open a pause menu dialog that lets you select "Main Menu", "Level Map", and "Component Factory" or "Back to Level" (depending on which one you're currently on). This would resolve 1, 4, and 5, if my formatting works the way I expect it to

Gameplay
  • Once you make the ALU, you're unable to test it again on its level, as it converts that level to a component factory schematic. When it was time to add the ALU to my computer, I wanted to refresh myself on how the ALU worked and not being able to test it made that more difficult. This issue is compounded by my next point...
  • Once the "enable" bit was added to Input I/O nodes, I was unable to manually set the value of the Input by flipping the bits on the left side of my screen. This made testing my circuits a huge pain, especially considering my next point...
  • You cannot step past a failure. When my circuit fails a test, I'd like to be able to step through and see what other tests it fails, that way I can get a better sense of how my circuit is behaving and attempt to address the failing tests all at once rather than one at a time. This is my biggest complaint right now
  • The 3-bit decoder challenge can very easily be cheesed with Add, Register, Mux, and On components, pictured here. This could easily be fixed by randomizing the order of the tests, resetting registers between tests, or disabling one of those components for that challenge.
  • In general, I worry that the tests in many levels aren't comprehensive enough to ensure that assembling all of the components you've built would actually result in a functioning computer. It's one thing if you're intentionally cheesing levels, like above, but the fact that it's possible to cheese a level like that so easily makes me wonder if I've come to the wrong solution on accident in other levels. I would look to Zachtronic games for an idea of what comprehensive tests with a good UI/UX looks like
